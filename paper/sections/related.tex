\section{Related Work}\label{sec:related}

\paragraph{Threshold automata and counter abstraction.}
Threshold automata were introduced by Konnov, Veith, and
Widder~\cite{KonnovVW2015} as a framework for parameterized verification
of fault-tolerant distributed algorithms.  The short counterexample
property~\cite{KonnovLVW2017} showed that safety violations can always
be witnessed within a bounded number of steps, enabling BMC-based
verification.  The ByMC tool~\cite{ByMC2017, KonnovWidder2018}
demonstrated this approach on simplified BFT protocols, and Bertrand
et~al.~\cite{BertrandKLW2021} extended it with decomposition techniques.
\tarsier{} builds on this foundation but addresses three limitations:
classical counter abstraction loses per-process delivery semantics
(addressed by our mode hierarchy, Sect.~\ref{sec:semantics}); ByMC has
no support for cryptographic objects (addressed by our certificate
declarations, Sect.~\ref{sec:crypto}); and ByMC requires trusting the
full toolchain (addressed by our proof certificates,
Sect.~\ref{sec:certificates}).

\paragraph{Explicit-state model checking.}
SPIN~\cite{Holzmann2003} and TLC for TLA+~\cite{LamportTLA2002}
enumerate concrete configurations, yielding results that hold only for
the fixed $n$ used during verification.  \tarsier{} keeps $n$, $t$,
and $f$ symbolic throughout, producing proofs for all instantiations
satisfying the resilience condition.  SPIN and TLA+ can model richer
state spaces (arbitrary data types, unbounded buffers) that fall
outside the threshold-automata fragment.

\paragraph{Interactive verification.}
Ivy~\cite{PadonMPSS2016} combines first-order logic with interactive
generalization; I4~\cite{MaGJKKS2019} automates invariant inference
via incremental learning.  Both handle general-purpose protocols but
require significant user guidance.  \tarsier{} targets the restricted
class of threshold-guarded protocols, where the short counterexample
property enables fully automatic verification without user-supplied
invariants---a deliberate trade-off that covers the core safety
arguments of PBFT, HotStuff, Tendermint, Paxos, and Algorand.

\paragraph{SMT-based model checking.}
BMC via SAT~\cite{BiereCCSZ1999}, $k$-induction~\cite{SheehySSS2000},
and PDR/IC3~\cite{Bradley2011} form the algorithmic core of
\tarsier{}'s engine.  We additionally implement
CEGAR~\cite{ClarkeGJLV2000} and portfolio solving across
Z3~\cite{DeMoura2008} and cvc5~\cite{BarbosaBBKLMMMN2022}.

\paragraph{Partial-order and symmetry reduction.}
Static POR~\cite{Peled1993, Godefroid1996} prunes independent
transitions.  \tarsier{} adapts POR to the counter-abstraction setting
and adds symmetry reduction for PDR cubes.

\paragraph{Proof-carrying verification.}
The proof-carrying code paradigm~\cite{Necula1997} separates proof
generation from validation.  \tarsier{} applies this to model checking:
the engine generates SMT-obligation bundles, validated by a minimal
kernel and optionally replayed against multiple solvers.  Solver proof
objects can be further checked via Carcara~\cite{Carcara2023}.

\paragraph{Probabilistic committee analysis.}
Algorand~\cite{GiladHMVZ2017, Micali2017} uses cryptographic sortition
for committee selection.  \tarsier{} computes the maximum adversary
bound~$b_{\max}$ via exact hypergeometric distributions with
arbitrary-precision arithmetic, injecting it into the SMT encoding for
a probabilistically safe verdict with quantified failure probability.
