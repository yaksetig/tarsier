\section{The \tarsier{} Specification Language}\label{sec:language}

\tarsier{} provides a domain-specific language (DSL) for specifying
distributed protocols.  The language is designed to closely resemble the
pseudo-code found in protocol papers while being formal enough for
automated verification.  A complete specification lives in a single
\texttt{.trs} file and is automatically lowered to a threshold automaton.

\subsection{Protocol Structure}

A specification declares symbolic parameters, a resilience condition, an
adversary model, message types, process roles, and properties to verify.
Figure~\ref{fig:rb-spec} shows Bracha's reliable
broadcast~\cite{Bracha1987} as a running example.

\begin{figure}[t]
\begin{lstlisting}
protocol ReliableBroadcast {
  params n, t, f;
  resilience: n > 3*t;
  adversary { model: byzantine; bound: f; }

  message Init();
  message Echo();
  message Ready();

  role Process {
    var decided: bool = false;
    init waiting;

    phase waiting {
      when received >= 1 Init() =>
        { send Echo(); goto phase echoed; }
    }
    phase echoed {
      when received >= 2*t+1 Echo() =>
        { send Ready(); goto phase readied; }
    }
    phase readied {
      when received >= 2*t+1 Ready() =>
        { decided = true; goto phase done; }
    }
    phase done {}
  }

  property agreement: agreement {
    forall p: Process. forall q: Process.
      p.decided == q.decided
  }
}
\end{lstlisting}
\caption{Bracha reliable broadcast in the \tarsier{} DSL.  The
\texttt{resilience} clause defines the fault model; \texttt{phase}
blocks define the process template; threshold guards use symbolic
expressions over parameters.}
\label{fig:rb-spec}
\end{figure}

The structure maps naturally to threshold automata: each phase becomes
a location, each \texttt{when received} clause becomes a rule with a
threshold guard, and the \texttt{resilience} clause becomes the
resilience condition~$\Phi$.

\subsection{Adversary Model}

The \texttt{adversary} block provides fine-grained control over the
threat model:
\begin{itemize}
  \item \textbf{Fault model:} \texttt{byzantine}, \texttt{crash}, or
    \texttt{omission}---determines which behaviors the adversary can
    exhibit.
  \item \textbf{Bound:} a parameter expression bounding the number of
    faulty processes (e.g.\ $f \leq t$).
  \item \textbf{Timing:} \texttt{async} or
    \texttt{partial\_synchrony}---the latter introduces a Global
    Stabilization Time (GST) after which all messages are delivered.
  \item \textbf{Authentication:} \texttt{signed} or \texttt{none}---when
    signed, sender identities are cryptographically bound to messages.
  \item \textbf{Network mode:} one of the four abstraction modes
    described in Sect.~\ref{sec:semantics}.
  \item \textbf{Equivocation:} \texttt{full} (the adversary may send
    conflicting messages to different recipients) or \texttt{none}
    (each sender commits to a single payload per round).
\end{itemize}

\subsection{Cryptographic Objects}\label{sec:crypto}

Modern BFT protocols (HotStuff~\cite{YinMGRGA2019},
Tendermint~\cite{BuchmanKM2018}) rely heavily on \emph{quorum
certificates} (QCs) and \emph{threshold signatures}.  \tarsier{} treats
these as first-class language constructs:

\begin{lstlisting}
certificate QuorumCert
  from Prepare threshold 2*t+1
  [signer Replica]
  [conflicts exclusive];

threshold_signature TSig signer Replica;
\end{lstlisting}

A \texttt{certificate} declaration specifies:
\begin{enumerate}
  \item the source message whose accumulation triggers formation;
  \item the threshold of \emph{distinct signers} required (a linear
    expression over parameters);
  \item an optional \texttt{conflicts exclusive} clause that prevents
    conflicting certificate variants from coexisting in an honest
    execution.
\end{enumerate}

The lowering pass translates these declarations into three families of
SMT constraints:
\begin{itemize}
  \item \textbf{Non-forgeability:}
    $\adv[k, \gamma_{\mathit{crypto}}] = 0$---the adversary
    cannot inject forged certificates.
  \item \textbf{Distinct-signer threshold:}
    Formation requires contributions from $\geq\!\tau$ \emph{distinct}
    sender identities, encoded (in identity-selective mode) as
    $\sum_s \mathit{ite}(\gamma_s > 0, 1, 0) \geq \tau$.
  \item \textbf{Conflict exclusion:}
    \texttt{conflicts exclusive} generates mutual-exclusion constraints
    between certificate variants in the same round, ruling out
    equivocation-based attacks.
\end{itemize}

\subsection{Lowering to Threshold Automata}\label{sec:lowering}

The lowering pass transforms a DSL specification into a threshold
automaton~$\ta$ in four steps:
\begin{enumerate}
  \item \textbf{Location expansion:}
    Each (role, phase, local-variable valuation) triple becomes a
    location.  For a role with $p$~phases and local variables with
    domain sizes $d_1, \ldots, d_k$, this produces
    $p \cdot \prod_i d_i$ locations.
  \item \textbf{Rule generation:}
    Each guarded transition becomes one or more rules with threshold
    guards derived from the expression and the active network mode.
  \item \textbf{Shared variable allocation:}
    Each (message family, recipient role, field valuation) tuple
    produces a shared variable.  Network modes refine this scoping
    (Sect.~\ref{sec:semantics}).
  \item \textbf{Property extraction:}
    Agreement properties become \emph{conflicting pairs}: pairs of
    locations corresponding to incompatible decisions that must not be
    simultaneously occupied.
\end{enumerate}

The lowering is deterministic and produces a well-formed threshold
automaton ready for the verification strategies of
Sect.~\ref{sec:verification}.
