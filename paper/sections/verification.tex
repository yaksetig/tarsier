\section{Verification Engine}\label{sec:verification}

\tarsier{} translates threshold automata into quantifier-free linear
integer arithmetic (QF\_LIA) formulae and applies BMC, $k$-induction,
and PDR to produce verdicts.  This section describes the encoding,
key optimizations, and the proof-certificate architecture.

\subsection{SMT Encoding}

Given a threshold automaton $\ta$ with locations~$\mathcal{L}$, shared
variables~$\Gamma$, rules~$\mathcal{R}$, and parameters~$\Pi$, the
encoder produces a QF\_LIA formula for each unrolling depth~$k$.

\paragraph{Variables.}
For each step $i \in [0, k]$ the encoding introduces:
\begin{itemize}
  \item $\kappa_i[\ell] \geq 0$ for each location~$\ell$
    (process counts);
  \item $\gamma_i[g] \geq 0$ for each shared variable~$g$;
  \item $\delta_i[r] \geq 0$ for each rule~$r$
    (processes firing $r$ at step~$i$);
  \item $\adv_i[g] \geq 0$ for each shared variable~$g$
    (adversary injection at step~$i$).
\end{itemize}
Parameters $n, t, f$ are declared as symbolic integer constants
constrained by the resilience condition~$\Phi$.

\paragraph{Initial state.}
All processes start in initial locations with all counters at zero:
\begin{equation}
  \bigwedge_{\ell \in \mathcal{I}} \kappa_0[\ell] = n
  \;\;\wedge\;\;
  \bigwedge_{\ell \notin \mathcal{I}} \kappa_0[\ell] = 0
  \;\;\wedge\;\;
  \bigwedge_{g} \gamma_0[g] = 0
\end{equation}

\paragraph{Step transition.}
For each step~$i$, the encoder asserts:
\begin{enumerate}
  \item \emph{Guard satisfaction:}
    $\delta_i[r] > 0 \Rightarrow g_r(\vec\gamma_i, \vec\pi)$---a rule
    fires only when its guard holds.
  \item \emph{Counter update:}
    Eq.~\eqref{eq:kappa-update} from
    Sect.~\ref{sec:counter-abstraction}.
  \item \emph{Shared variable update:}
    $\gamma_{i+1}[g] = \gamma_i[g] + \sum_r u_r^g \cdot \delta_i[r]
    + \adv_i[g]$,
    where $u_r^g \in \{0, 1\}$ is the increment specified by rule~$r$.
  \item \emph{Feasibility:}
    $\sum_{\{r : \mathit{src}(r)=\ell\}} \delta_i[r] \leq
    \kappa_i[\ell]$---cannot fire more processes than are available.
  \item \emph{Adversary bound:}
    $\sum_{i=0}^{k} \adv_i[g] \leq f$---total injection is bounded
    by the fault parameter.
\end{enumerate}

\paragraph{Safety property.}
For agreement, the encoder generates a bad-state disjunction:
$\bigvee_{(a,b) \in C} (\kappa_k[a] > 0 \wedge \kappa_k[b] > 0)$,
where $C$ is the set of conflicting location pairs from lowering
(Sect.~\ref{sec:lowering}).  The formula is satisfiable iff a bad
state is reachable within $k$~steps.

\subsection{Optimizations}\label{sec:optimizations}

\paragraph{Structural hashing.}
The encoder canonicalizes SMT terms via string keys with canonical
ordering for commutative operators, eliminating duplicate assertions.
This typically reduces the assertion count by 30--70\%.

\paragraph{Incremental solving.}
For $k$-induction, the solver state is extended from depth~$k$ to
$k{+}1$ without re-encoding previous steps.  Assertions from earlier
depths are reused via incremental push/pop scopes, amortizing encoding
cost.

\paragraph{Partial-order reduction (POR).}
\tarsier{} implements both static POR (pruning commuting
rules~\cite{Peled1993, Godefroid1996}) and dynamic ample sets during
PDR (disabling rules whose guards are independent of the current
cube's constrained variables).

\paragraph{Symmetry reduction.}
In PDR, cubes that are symmetric under parameter permutation are
deduplicated, pruning 10--40\% of redundant candidates.

\subsection{Proof Certificates}\label{sec:certificates}

To address the trust challenge from Sect.~\ref{sec:intro}, \tarsier{}
generates \emph{proof certificates} that can be validated independently
of the verification engine.

\paragraph{Certificate structure.}
A certificate bundle contains:
\begin{itemize}
  \item A \emph{manifest} with the protocol source hash, engine
    parameters, solver version, and timestamp.
  \item The \emph{SMT obligations}---the formulae submitted to the
    solver---together with each solver's verdict (UNSAT for valid
    proofs).
  \item SHA-256 hashes of each obligation, and a domain-tagged bundle
    hash covering all metadata.
  \item Optionally, \emph{proof objects} in the Alethe format emitted
    by the solver.
\end{itemize}

The obligations depend on the proof engine:
\begin{itemize}
  \item \textbf{$k$-Induction:} base case (BMC up to~$k$) and
    inductive step.
  \item \textbf{PDR:}
    $\mathit{init} \Rightarrow \mathit{inv}$,\;
    $\mathit{inv} \wedge \mathit{trans} \Rightarrow \mathit{inv}'$,\;
    $\mathit{inv} \Rightarrow \mathit{safe}$.
\end{itemize}

\paragraph{Minimal proof kernel.}
The \texttt{tarsier-proof-kernel} validates certificate bundles with
\emph{only four library dependencies} (\texttt{sha2}, \texttt{serde},
\texttt{serde\_json}, \texttt{thiserror}).  It shares no code with the
verification engine and checks:
\begin{enumerate}
  \item Schema version compatibility;
  \item Obligation integrity via SHA-256;
  \item Bundle-level hash via a domain-tagged construction;
  \item Obligation completeness for the declared proof engine;
  \item Path safety (no path-traversal attacks in file references).
\end{enumerate}

\paragraph{Multi-solver replay.}
The \texttt{tarsier-certcheck} tool replays each obligation against
multiple solvers (e.g.\ Z3 and cvc5).  A \emph{reinforced} assurance
level requires agreement from at least two independent solvers.
A \emph{high-assurance} level additionally validates solver proof
objects via the Carcara proof checker~\cite{Carcara2023}.
