# Reconfiguration/Epoch Semantics Gap Report

> Generated by RECONF-01 audit of the DSLв†’IRв†’SMT pipeline for `reconfigure` actions.

> Status update (2026-03-10): this document is historical. Current `main` re-asserts
> resilience in epoch-aware BMC/k-induction encoders and adds strict-mode fail-closed
> preflight checks that reject reconfigure updates that can weaken resilience (for example,
> `t = t + 1` under `n > 3*t`). The original GAP-1 reproducer now fails validation in
> strict mode.

## Executive Summary

The reconfiguration pipeline (DSL parsing в†’ IR lowering в†’ SMT encoding) is **partially
complete**. The core mechanics work: parameters are marked time-varying, per-step SMT
variables are generated, and frame/update constraints are emitted. However, the audit
identified **6 concrete gaps**, 2 of which are **soundness-critical** for protocols that
use `reconfigure` actions.

| Gap | Severity | Category | Effort |
|-----|----------|----------|--------|
| GAP-1: Resilience not re-asserted after epoch | Critical | Soundness | Medium |
| GAP-2: Multi-rule same-param conflict | High | Soundness (conditional) | Medium |
| GAP-3: `ReconfigurationSpec` never populated by lowering | High | Missing wiring | Low |
| GAP-4: `ReconfigurationSemantics` not read by encoder | High | Incomplete feature | Medium |
| GAP-5: `max_reconfigurations` not enforced | Medium | Incomplete feature | Medium |
| GAP-6: k-induction encoder lacks epoch support | Medium | Incomplete feature | Medium-High |

## GAP-1: Resilience Condition Not Re-Asserted After Epoch Transitions (CRITICAL)

### Description

The resilience condition (e.g., `n > 3*t`) is encoded **only once** at initialization,
using global parameter variables `p_i`. After a `reconfigure` action changes a time-varying
parameter at step `k`, the new value `p_i_{k+1}` is **never checked** against the
resilience condition.

### Files

- `crates/tarsier-smt/src/encoder/mod.rs:252-265` вЂ” resilience asserted once with global `p_i`
- `crates/tarsier-smt/src/encoder/k_induction.rs:125-137` вЂ” same pattern

### Impact

A protocol can reconfigure itself into a state where the resilience condition is violated,
and the solver will not detect it. This is a **soundness issue**: the BMC/k-induction
proof assumes `n > 3*t` holds throughout execution, but reconfiguration can break this
invariant silently.

### Reproducer

The existing `reconfig_threshold_buggy.trs` demonstrates this partially вЂ” it weakens `t`
via `reconfigure { t = t - 1; }`. However, the following reproducer explicitly **breaks
resilience**:

```trs
// reproducer_gap1_resilience_break.trs
// After reconfigure, t becomes larger than n/3, violating resilience.
// The solver should report UNSAFE but currently cannot detect the violation
// because resilience is only checked at initialization.
protocol ResilienceBreak {
    params n, t, f;
    resilience: n > 3*t;
    adversary { model: byzantine; bound: f; }
    message Vote;
    role Process {
        var decided: bool = false;
        init waiting;
        phase waiting {
            when received >= 1 Vote => {
                reconfigure {
                    t = t + 1;  // Breaks n > 3*t after one step
                }
                decided = true;
                decide true;
                goto phase done;
            }
        }
        phase done {}
    }
    property agreement: agreement {
        forall p: Process. forall q: Process.
            (p.decided == true && q.decided == true) ==> (p.decision == q.decision)
    }
}
```

### Recommended Fix (RECONF-02)

After emitting `p_i_{k+1}` update/frame constraints, re-assert the resilience condition
using `p_i_{k+1}` variables:

```rust
// In phase_encode_transition_relation (mod.rs, after line 1121):
if !time_varying_param_ids.is_empty() {
    if let Some(ref rc) = ta.constraints.resilience_condition {
        let lhs = encode_lc_at_step(&rc.lhs, k + 1, time_varying_param_ids);
        let rhs = encode_lc_at_step(&rc.rhs, k + 1, time_varying_param_ids);
        let constraint = match rc.op {
            CmpOp::Gt => lhs.gt(rhs),
            // ... other operators
        };
        enc.assert_term(constraint);
    }
}
```

## GAP-2: Multiple Rules Updating Same Parameter вЂ” No Conflict Resolution (HIGH)

### Description

When two or more rules update the same parameter at the same step, the encoder generates
independent implications:

```
delta_r1 > 0 в†’ p_{k+1} = value1(params_k)
delta_r2 > 0 в†’ p_{k+1} = value2(params_k)
```

If both rules fire (`delta_r1 > 0` and `delta_r2 > 0`) with `value1 в‰  value2`, the
constraints become contradictory. Depending on the protocol structure, this can either:
- Make steps where both rules fire unsatisfiable (masking bugs at those depths), or
- Still find bugs via steps where only one rule fires.

**Empirical finding:** The reproducer protocol finds UNSAFE at depth 3 via BMC, because
the two rules involve different source/target locations and the solver can find a path
where only one fires. However, for protocols where conflicting update rules share
locations, the contradictory constraints could mask bugs entirely.

### Files

- `crates/tarsier-smt/src/encoder/mod.rs:1092-1119` вЂ” no mutual exclusion between update rules

### Impact

Any protocol with multiple transitions that reconfigure the same parameter can produce
false-safe results. The BMC encoding becomes over-constrained because conflicting updates
make the transition relation unsatisfiable.

### Reproducer

```trs
// reproducer_gap2_multi_update_conflict.trs
// Two rules update t to different values.
// Both can fire in the same step, creating contradictory SMT constraints.
// The solver should find this UNSAFE (trivial disagreement),
// but the conflicting parameter update makes the formula UNSAT.
protocol MultiUpdateConflict {
    params n, t, f;
    resilience: n > 3*t;
    adversary { model: byzantine; bound: f; }
    message Vote;
    message Reject;
    role Process {
        var decided: bool = false;
        var decision: bool = false;
        init waiting;
        phase waiting {
            when received >= 1 Vote => {
                reconfigure { t = 1; }
                decision = true;
                decided = true;
                decide true;
                goto phase done_yes;
            }
            when received >= 1 Reject => {
                reconfigure { t = 2; }
                decision = false;
                decided = true;
                decide false;
                goto phase done_no;
            }
        }
        phase done_yes {}
        phase done_no {}
    }
    property agreement: agreement {
        forall p: Process. forall q: Process.
            (p.decided == true && q.decided == true) ==> (p.decision == q.decision)
    }
}
```

### Recommended Fix (RECONF-02)

Add a mutual-exclusion constraint when multiple rules update the same parameter:

```rust
// Option A: At most one updating rule fires per step (strict)
if update_rules.len() > 1 {
    let fire_vars: Vec<SmtTerm> = update_rules
        .iter()
        .map(|&(r, _)| SmtTerm::var(delta_var(k, r)).gt(SmtTerm::int(0)).ite(SmtTerm::int(1), SmtTerm::int(0)))
        .collect();
    let total = sum_terms_balanced(fire_vars);
    enc.assert_term(total.le(SmtTerm::int(1)));
}
```

```rust
// Option B: Use if-then-else chain with priority (more flexible)
// First firing rule wins; later rules only apply if earlier ones don't fire.
```

## GAP-3: `ReconfigurationSpec` Never Populated by Lowering

### Description

The IR defines `ReconfigurationSpec` (with `semantics` and `max_reconfigurations` fields)
and `ThresholdAutomaton` has a `reconfiguration: Option<ReconfigurationSpec>` field. However,
the lowering code **never creates or assigns** a `ReconfigurationSpec`, even when
`reconfigure` actions are present. The field is always `None`.

### Files

- `crates/tarsier-ir/src/threshold_automaton.rs:381` вЂ” `pub reconfiguration: Option<ReconfigurationSpec>`
- `crates/tarsier-ir/src/lowering/mod.rs` вЂ” no `ReconfigurationSpec` construction anywhere

### Impact

The `ReconfigurationSpec` is dead infrastructure. The encoder cannot distinguish
`NextStep` vs `Immediate` semantics or enforce `max_reconfigurations` because the spec
is never populated.

### Recommended Fix (RECONF-02)

In the lowering pass, after marking time-varying parameters (line ~1530), detect whether
any reconfigure actions exist and populate the spec:

```rust
if ta.has_reconfiguration() {
    ta.reconfiguration = Some(ReconfigurationSpec {
        semantics: ReconfigurationSemantics::NextStep,  // default
        max_reconfigurations: 0,  // unbounded by default
    });
}
```

Future DSL extension: allow `reconfigure_semantics: immediate;` at protocol level.

## GAP-4: `ReconfigurationSemantics` Not Read by SMT Encoder

### Description

The SMT encoder (both BMC and k-induction) never reads `ta.reconfiguration` or checks
`ReconfigurationSemantics`. The current encoding **implicitly implements `NextStep`
semantics**: the update formula `p_{k+1} = value(params_k)` uses old-step parameters,
and the new values are first used at step `k+1`.

However, `Immediate` semantics is not implementable without code changes.

### Files

- `crates/tarsier-smt/src/encoder/mod.rs` вЂ” zero references to `ReconfigurationSemantics`
- `crates/tarsier-smt/src/encoder/k_induction.rs` вЂ” zero references

### Impact

Low (currently), since only `NextStep` is expected. But the `Immediate` variant exists
in the type system and could mislead users who set it manually on the IR.

### Recommended Fix (RECONF-02)

At minimum, add a runtime check in the encoder that rejects `Immediate` semantics
with a clear error message until it's implemented. This prevents silent misuse.

## GAP-5: `max_reconfigurations` Not Enforced in Encoding

### Description

The `ReconfigurationSpec::max_reconfigurations` field exists in the IR but:
1. There's no DSL syntax to set it
2. The encoder doesn't generate counting constraints
3. The field is never populated (see GAP-3)

### Files

- `crates/tarsier-ir/src/threshold_automaton.rs:1606` вЂ” field definition only

### Impact

Medium. Without a bound on reconfigurations, the BMC unrolling explores all possible
reconfiguration sequences up to depth `k`. For protocols with complex reconfiguration
patterns, this could lead to state space explosion. More importantly, some protocols
are only safe under bounded reconfiguration.

### Recommended Fix (RECONF-02)

When `max_reconfigurations > 0`, introduce a reconfiguration counter:

```rust
// Declare: reconf_count_0 = 0
// At each step k: reconf_count_{k+1} = reconf_count_k + (1 if any reconf rule fires, else 0)
// Assert: reconf_count_{k+1} <= max_reconfigurations
```

## GAP-6: K-Induction Encoder Lacks Epoch Support

### Description

The k-induction encoder (`k_induction.rs`) declares parameters as global `p_i` only.
It does not:
1. Detect time-varying parameters
2. Declare per-step parameter variables
3. Encode parameter update/frame constraints
4. Use epoch-aware guard encoding

### Files

- `crates/tarsier-smt/src/encoder/k_induction.rs:116-137` вЂ” parameters only global
- Compare with `crates/tarsier-smt/src/encoder/mod.rs:245-250` вЂ” BMC encoder has epoch support

### Impact

K-induction proofs for protocols with `reconfigure` actions are **unsound**: they prove
safety assuming parameters never change, which is false. The BMC base-case encoding
correctly handles epochs, but the induction step does not.

### Recommended Fix (RECONF-02)

Mirror the BMC encoder's epoch handling in the k-induction encoder:
1. Detect `time_varying_param_ids` from the counter system
2. Declare `p_i_k` variables for the induction step
3. Encode parameter update/frame constraints
4. Use `encode_threshold_guard_at_step_epoch` for guards
5. Re-assert resilience at each step boundary

## Conclusion

The core reconfiguration mechanics (DSL parsing, lowering, basic SMT variable management)
are solid. The critical gaps are in **constraint completeness**: the solver trusts that
parameters remain valid after reconfiguration, which is not enforced. GAP-1 and GAP-2
should be fixed before any protocol with `reconfigure` actions is considered verified.
GAP-6 (k-induction) should be fixed before unbounded proofs are trusted for such protocols.
