# Engine Panic/Unwrap Inventory and Remediation Plan

> Generated by PANIC-01 audit of `tarsier-engine` non-test production code.

## Executive Summary

The `tarsier-engine` crate has **excellent error handling discipline**. A comprehensive
audit found only **3 panic-family calls** in production (non-test) code out of ~34,000
lines. All three are defensive assertions with sound guards, not open-ended crash risks.

The initial estimate of "55+ panics" was incorrect — it counted test code (`#[cfg(test)]`
modules and dedicated test files like `tests.rs`, `property_pipeline_unit_tests.rs`),
which correctly use `unwrap()`/`panic!()` for assertion clarity.

---

## Production Code Inventory

### 1. `pipeline/analysis.rs:516` — `expect("len() checked")`

**Severity:** Low (guard ensures non-empty)

```rust
if candidate_params.len() == 1 {
    let pid = *candidate_params.iter().next().expect("len() checked");
    ta.constraints.adversary_bound_param = Some(pid);
}
```

**Context:** `analyze_and_constrain_committees()` selects the adversary bound parameter
from a candidate set. The `.len() == 1` guard guarantees `.next()` returns `Some`.

**Remediation (PANIC-02):**
```rust
if candidate_params.len() == 1 {
    if let Some(&pid) = candidate_params.iter().next() {
        ta.constraints.adversary_bound_param = Some(pid);
    }
}
```

**Risk if not fixed:** Zero — the guard is correct. The `expect` is cosmetically
suboptimal but will never fire.

### 2. `pipeline/verification/smt_helpers.rs:27` — `expect("SAT should include model")`

**Severity:** Medium (depends on solver contract)

```rust
let (result, model) = solver.check_sat_with_model(&var_refs)?;
Ok(match result {
    SatResult::Sat => BmcResult::Unsafe {
        depth,
        model: model.expect("SAT should include model"),
    },
    SatResult::Unsat => BmcResult::Safe { depth_checked: depth },
    SatResult::Unknown(reason) => BmcResult::Unknown { depth, reason },
})
```

**Context:** `run_single_depth_bmc_encoding()` relies on the solver returning a model
when the result is `Sat`. The Z3 crate (v0.19) always returns a model with `Sat`, but
an alternative solver backend could violate this contract.

**Remediation (PANIC-03):**
```rust
SatResult::Sat => {
    let model = model.ok_or_else(|| PipelineError::SolverContract(
        "SAT result returned without model".into()
    ))?;
    BmcResult::Unsafe { depth, model }
}
```

**Risk if not fixed:** Low with Z3-only; medium if cvc5 or other backends are added
without the same model guarantee.

### 3. `pipeline/verification/cegar.rs:1435` — `unreachable!("match arm already covered")`

**Severity:** Very Low (genuinely unreachable by construction)

```rust
UnboundedSafetyResult::NotProved { .. } | UnboundedSafetyResult::Unknown { .. } => {
    let reason = match result {
        UnboundedSafetyResult::NotProved { max_k, .. } => {
            format!("proof did not close up to k={max_k}")
        }
        UnboundedSafetyResult::Unknown { reason } => reason.clone(),
        _ => unreachable!("match arm already covered by outer pattern"),
    };
```

**Context:** The outer match arm restricts `result` to `NotProved | Unknown`. The inner
match exhaustively handles both variants, making the `_ =>` branch dead code.

**Remediation (PANIC-03):**
No change needed. The `unreachable!()` is correct defensive programming. If the enum
gains new variants, the compiler will warn at the outer match first. Optionally, this
could be replaced with `_ => String::new()` to avoid the macro, but that would hide
logic errors rather than surfacing them.

**Risk if not fixed:** Zero.

## Test Code (excluded from remediation)

Test files contain ~170+ `unwrap()`/`expect()`/`panic!()` calls, which is normal
and expected for assertion-heavy test code. These are properly isolated behind
`#[cfg(test)]` and in dedicated test files:

- `pipeline/property/tests.rs` (~90 calls)
- `pipeline/verification/tests.rs` (~5 calls)
- `property_pipeline_unit_tests.rs` (~10 calls)
- Various `#[cfg(test)]` modules in production files (~65 calls)

No remediation needed for test code.

## Remediation Roadmap

| Task | Scope | Effort |
|------|-------|--------|
| **PANIC-02** | Replace `expect` in `analysis.rs:516` with `if let` pattern | Trivial (1 line) |
| **PANIC-03** | Replace `expect` in `smt_helpers.rs:27` with `Result` propagation | Low (add error variant + 3 lines) |
| **PANIC-04** | Verify no regressions after PANIC-02/03 fixes | Low (run existing tests) |
| **PANIC-05** | Add CI lint banning `panic!/unwrap/expect` in engine non-test code | Medium (clippy config + allowlist) |

## Conclusion

The engine crate is production-ready from an error-handling perspective. The 3 findings
are cosmetic improvements, not crash risks. The highest-value follow-up is PANIC-03
(solver contract `expect`), which would improve robustness for multi-solver scenarios.
